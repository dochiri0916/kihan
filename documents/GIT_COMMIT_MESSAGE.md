다음 규칙을 반드시 따른다.

1. 커밋 메시지는 Conventional Commits 1.0.0 표준을 따른다.
2. 커밋 메시지는 반드시 다음 형식으로 작성한다.
   <타입>: <한국어 설명>
3. 타입은 아래 목록 중 하나만 사용한다.
   - feat: 새로운 기능 추가
   - fix: 버그 수정
   - refactor: 기능 변경 없는 리팩토링
   - perf: 성능 개선
   - test: 테스트 코드 추가 또는 수정
   - docs: 문서 수정
   - style: 포맷, 공백, 세미콜론 등 코드 스타일 변경
   - build: 빌드/의존성 관련 변경
   - ci: CI 설정 변경
   - chore: 기타 변경 사항
4. 설명은 반드시 한국어로 작성한다.
5. 설명은 한 줄 요약이며 명령형이나 과거형을 사용하지 않는다.
6. 불필요한 수식어 없이 변경의 핵심만 간결하게 작성한다.
7. 범위(scope)는 사용하지 않는다.
8. 하나의 커밋은 하나의 타입과 하나의 변경 목적만 가진다.
9. 서로 다른 타입의 변경은 반드시 분리된 커밋 메시지로 생성한다.
10. 여러 파일이 변경된 경우라도 타입과 목적이 동일할 때만 하나의 커밋 메시지로 묶을 수 있다.
11. 여러 타입의 변경이 존재하는데 커밋 메시지를 하나만 생성하는 것은 금지한다.
12. 커밋 메시지에는 반드시 변경 대상이 드러나야 한다.
13. 결과는 커밋 메시지만 출력한다.
14. 여러 커밋이 필요한 경우 커밋 메시지를 여러 줄로 출력한다.
