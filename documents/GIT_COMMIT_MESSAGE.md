다음 규칙을 반드시 따른다.

1. 커밋 메시지는 Conventional Commits 1.0.0 표준을 따른다.
2. 커밋 메시지는 반드시 다음 형식으로 작성한다.
   <타입>: <한국어 설명>
3. 타입은 아래 목록 중 하나만 사용한다.
   - feat: 새로운 기능 추가
   - fix: 버그 수정
   - refactor: 기능 변경 없는 리팩토링
   - perf: 성능 개선
   - test: 테스트 코드 추가 또는 수정
   - docs: 문서 수정
   - style: 포맷, 공백, 세미콜론 등 코드 스타일 변경
   - build: 빌드/의존성 관련 변경
   - ci: CI 설정 변경
   - chore: 기타 변경 사항
4. 설명은 반드시 한국어로 작성한다.
5. 설명은 한 줄 요약이며 명령형이나 과거형을 사용하지 않는다.
6. 불필요한 수식어 없이 변경의 핵심만 간결하게 작성한다.
7. 범위(scope)는 사용하지 않는다. (예: feat(auth) 금지)
8. 여러 파일이 변경된 경우 원칙적으로 각 파일마다 하나의 커밋 메시지를 생성한다.
   단, 동일한 타입(type)이며 동일한 목적의 변경인 경우에만 하나의 커밋 메시지로 묶을 수 있다.
   서로 다른 타입의 변경(feat, docs, refactor 등)은 반드시 분리한다.
9. 커밋 메시지에는 반드시 변경 대상이 드러나야 한다.
   (예: 엔티티 필드 추가, 서비스 로직 수정, 설정 변경, 문서 항목 정리)
10. 결과는 커밋 메시지 한 줄만 출력한다.

이제 변경 내용을 보고 커밋 메시지를 생성하라.
